# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Stack

- **Framework**: Next.js 14 App Router (React Server Components + Route Handlers)
- **Language**: TypeScript
- **Styling**: Tailwind CSS v3 + custom design tokens (gold `#C9A84C`, dark backgrounds `#0A0A0A / #141414 / #1a1a1a`)
- **Database**: Supabase (PostgreSQL) — accessed via `@supabase/supabase-js` (service role) or `@supabase/ssr` (session-aware)
- **Auth**: Supabase Auth — session read via `@supabase/ssr` in RSC/Route Handlers; roles stored in `public.users.role`
- **PDF generation**: `pdfkit` (pure Node.js — replaces `@react-pdf/renderer` which conflicts with Next.js RSC JSX runtime)
- **Email**: Resend (`src/lib/resend.ts`)
- **i18n**: `next-intl` — locales `es` (default) and `en`, messages in `messages/es.json` + `messages/en.json`
- **Maps**: Mapbox GL JS (`NEXT_PUBLIC_MAPBOX_TOKEN`)
- **Forms**: `react-hook-form` + `zod`

## Architecture

### Route structure
```
src/app/[locale]/          — All pages (locale = es | en)
  page.tsx                 — Home (ticker, map, stats)
  admin/                   — Admin panel (layout.tsx guards role === "admin")
    sellers/[id]/          — Seller detail: KYC docs + status + traceability
    buyers/[id]/           — Buyer detail
    documents/             — Generated docs list + send form + template editor
    signatories/           — DZ Metals signatories + signature image upload
  portal/                  — Client portal (layout.tsx guards authenticated user)
    documents/             — Client's received docs + upload signed version
  register/seller|buyer/   — Public 3-step registration forms

src/app/api/
  admin/documents/send         — Generate PDF, upload to Storage, insert DB record
  admin/documents/[id]/pdf     — Signed URL redirect to PDF in Storage
  admin/documents/[id]/signed  — Download client-signed doc + mark completed
  admin/templates/[id]         — PATCH template content_es / content_en
  admin/users/[id]/status      — PATCH status (pending → active | rejected)
  admin/users/[id]/delete      — DELETE from DB + Supabase auth
  admin/signatories/signature  — POST signature PNG to `signatures` bucket
  portal/documents/[id]/sign   — POST client uploads signed doc to `signed-docs`
  register/seller|buyer        — Public registration: creates auth user + profile row
  metals-prices                — Proxies metals API or returns mock data
```

### Supabase client pattern — CRITICAL
**Never mix clients for auth and DB operations.**

| Use case | Client |
|---|---|
| Read session / `auth.getUser()` | `createServerClient` from `@supabase/ssr` with cookie store |
| Any DB or Storage operation in admin routes | `createClient(URL, SERVICE_ROLE_KEY)` — bypasses RLS |

Cookie `setAll` in Server Components must be wrapped in `try/catch` — setting cookies outside Route Handlers/Server Actions throws but is non-fatal.

### User roles & status
- Roles: `buyer`, `seller`, `admin` — stored in `public.users.role`
- Status: `pending` → `active` | `rejected` — stored in `public.users.status`
- **Use `"active"` not `"approved"`** — the DB and all code uses `"active"` for approved users

### Key database tables
| Table | Purpose |
|---|---|
| `users` | Mirrors Supabase auth; has `role`, `status`, `kyc_id_url`, `kyc_selfie_url` |
| `seller_profiles` | Material details; `traceability` enum: `completa\|incompleta\|no_tengo` |
| `buyer_profiles` | Buyer details |
| `document_templates` | 7 templates (`loi`, `nda`, `sco`, `ncnda`, `mandato_vendedor`, `mandato_comprador`, `compraventa`); content in `content_es`/`content_en` with `{{variable}}` markers; `requires_client_signature` boolean (SCO = false) |
| `generated_documents` | Issued docs; status: `sent→viewed→downloaded→signed→completed`; has `signed_document_path` + `signed_at` |
| `signatories` | DZ Metals signatories; `signature_url` = path in `signatures` bucket |

### Storage buckets (all private — always use signed URLs)
| Bucket | Contents |
|---|---|
| `kyc` | Seller/buyer identity documents |
| `signatures` | Signatory signature PNG images |
| `generated-docs` | PDFs generated by admin |
| `signed-docs` | Signed documents uploaded by clients |

### PDF generation (`src/lib/pdf/generateDocument.ts`)
Uses `pdfkit` imperatively (not React/JSX). Must be listed in `serverComponentsExternalPackages` in `next.config.mjs`. Main export: `generateDocumentPDF()` returns `Buffer`. Template `{{variable}}` markers are substituted via `fillTemplate()`. Do NOT reintroduce `@react-pdf/renderer` — it conflicts with the Next.js RSC JSX runtime.

### Document signing flow
1. Admin sends doc → PDF generated with `pdfkit` → uploaded to `generated-docs` → `generated_documents` row inserted
2. Client downloads PDF, prints, signs physically, scans
3. Client uploads scan → `signed-docs` bucket → status becomes `signed`
4. Admin downloads signed doc → status becomes `completed`

`requires_client_signature` on the template controls whether the "Subir firmado" button appears in the client portal.

### Ticker (`MetalsPriceTicker`)
Client component polling `/api/metals-prices` every 60s. Infinite scroll via `animate-ticker` keyframe in `tailwind.config.ts` (`translateX(-33.333%)` on a tripled item array). Falls back to mock data if `METALS_API_KEY` is unset.

### i18n
All pages under `src/app/[locale]/`. Always import `Link`, `redirect`, `usePathname`, `useRouter` from `src/navigation.ts` (next-intl wrappers) — not from Next.js directly.

## SQL migrations
No ORM migrations — schema managed manually. Reference DDL in `supabase/documents.sql`. Scripts in `scripts/` (e.g. `load-loi.js` loads template content). Always use `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for additive changes.
